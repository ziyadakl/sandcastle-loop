# ADR 0007 — Sandcastle: bump @ai-hero/sandcastle 0.5.10 → 0.7.0

**Status:** Accepted
**Date:** 2026-06-04

## Context

The repo pinned `@ai-hero/sandcastle ^0.5.10`; upstream had reached 0.7.0. This
repo is a customized fork that follows "Matt's pattern, not his stock code"
(CONTEXT.md): it runs its own orchestrator (`.sandcastle/main.mts`) and init
shim (`bin/init.mjs`), so upstream changes are either *in the engine* (free with
the bump) or *in stock templates / init flows this repo replaced* (N/A). There
is no shared git lineage with upstream; updates are hand-applied.

## Decision

**Bump to 0.7.0.** Verified safe before landing: typecheck clean, full suite
green (562 → 581 incl. new tests). No source changes were required.

Why no source changes:

- 0.7.0's BREAKING removal of the `SessionStore` public API does not touch us —
  it was never imported. The session-path logic we depend on (`encodeProjectPath`
  mirror in `lib/skill-discipline.ts`) is byte-identical to 0.7.0's, and the
  `~/.claude/projects/<encoded>/<id>.jsonl` convention (Claude Code's, not the
  SDK's) is unchanged. `IterationResult.sessionId/sessionFilePath` are preserved.
- 0.7.0 drops the bundled Effect runtime, so `package-lock.json` shed ~50
  transitive deps — a slimming win.

**Lockfile: npm.** `package-lock.json` is what `main` tracks and what upstream
declares (`packageManager: npm@10.9.2`). An in-flight pnpm migration (untracked
`pnpm-lock.yaml`/`pnpm-workspace.yaml`) is owned by a separate effort; updating
the npm lockfile is direction-agnostic and will not collide with it.

**Prompt audit — nothing to port.** Upstream's only 0.6→0.7 prompt changes (treat
an empty filtered task list as ground truth; discourage unfiltered issue
re-queries) live in Matt's stock templates we do not run. Audited every custom
prompt: only `plan-prompt.md` queries the issue list, and it already covers both
ideas more rigorously — rule 5 treats zero eligible issues as ground truth, and
the filtered `<issues-json>` vs unfiltered `<all-open-issues>` split (plus the
truncation-halt guard) is stricter than upstream's hint.

**Future-proofing.** Two low-risk safeguards added:

- A drift-guard test (`tests/skill-discipline.test.ts`) asserting our session-path
  encoding matches the SDK's now-public `claudeHostSessionPath`, so a future SDK
  encoding change fails loudly instead of the skill-discipline gate silently
  abstaining (the ADR 0006 failure class).
- A `pnpm sandcastle:check-upstream` helper reporting pinned-vs-latest version +
  the exact SDK symbols this repo imports, so the next bump starts from a
  targeted "search the changelog for these names" checklist.

## Considered but deferred

- **Adopt the SDK's now-public path helpers in `resolveSessionFilePath`** (drop the
  hand-mirror). Deferred: it changes the skill-discipline gate's *runtime*
  file-finding — the exact mechanism whose silent failure is ADR 0006's nightmare
  — and our tests are synthetic fixtures that would not catch a production
  regression. The drift-guard test gives the same protection with zero runtime
  change. Revisit only behind an integration test that exercises real session
  capture.
- **Auto-fetch + correlate the upstream changelog in the helper.** Deferred on
  YAGNI: the gain (skipping one manual changelog scan, on infrequent
  user-controlled upgrades) is tiny and rare; the cost (a parser coupled to an
  external changelog's format, which rots silently) is permanent. Add it only if
  the manual scan becomes a real pain.

## Consequences

- Engine upgraded with no behavioral change to the loop; dependency tree slimmer.
- The hand-mirror in `skill-discipline.ts` stays, now guarded against drift.
- Operators get a one-command upstream check for the next bump.
